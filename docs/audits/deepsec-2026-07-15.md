# DeepSec security audit — 2026-07-15

## Scope and method

- [DeepSec 2.2.1](https://github.com/vercel-labs/deepsec/) ran in a secret-free temporary copy of the current working tree.
- The scan used Codex with `gpt-5.6-sol` and `xhigh` reasoning for both analysis and revalidation.
- Static matching inspected 347 source files; agent analysis reviewed 150 candidate files.
- DeepSec revalidated all 25 initial findings as true positives. A separate manual review then checked every finding against the live working tree and Cloudflare's documented runtime behavior.
- The original audit was read-only. The 2026-07-17 remediation and revalidation are recorded below.

DeepSec's raw severity totals were 0 critical, 0 high, 7 medium, 8 high-impact bugs, and 10 bugs. Manual triage found 7 security vulnerabilities, 1 environment-dependent security risk, 16 non-security bugs, and 1 scanner false positive.

## Confirmed security vulnerabilities

1. **Stale public-page saves can restore revoked public access.** The dashboard submits a cached full replacement, while the Worker increments but does not compare the existing revision. A stale authorized tab can re-enable a page or restore recordings after another administrator revoked them. Evidence: `apps/dashboard/src/routes/settings/settings-public-page-card.tsx:63` and `apps/worker/src/api/public-page-settings.ts:98`. Proposed direction: include `expectedRevision`, use a compare-and-swap update, return `409` on conflict, and refetch before retrying.

2. **Replayed live tickets perform full snapshot work before nonce rejection.** One valid ticket can be replayed repeatedly during its lifetime, forcing the session Durable Object to scan stored metadata before returning `ticket_used`. Evidence: `apps/worker/src/do/session-live-hub.ts:129`. Proposed direction: atomically consume the nonce before snapshot construction and add a fail-closed live-upgrade attempt limit.

3. **Finalization expands the complete attacker-controlled metadata history in memory.** `finalPageBatches()` loads and parses every stored sidecar at once. The accepted 64 MiB metadata budget can expand into far more live JavaScript memory and repeatedly prevent finalization. Evidence: `apps/worker/src/do/session-recorder-store.ts:380`. Proposed direction: keyset-page analytics reconstruction, retain only bounded per-tab state, and enforce a finalization-safe event budget.

4. **Public ingest can exploit the finalization allocation and evade finalized-session quota tracking.** Browser-visible project write keys allow an attacker to build sessions that hit the memory failure above; those sessions never reach the D1 rows used for normal quota and retention work. Evidence: `apps/worker/src/do/session-recorder.ts:233` and `apps/worker/src/consumer/queue.ts:161`. Proposed direction: fix the streamed finalization path and account for accepted pending Durable Object/R2 storage before finalization.

5. **Live join enforces aggregate replay limits after retaining the decoded checkpoint window.** A crafted checkpoint-free session can make an operator's browser decode and retain substantially more than the intended 128 MiB limit before rejection. Evidence: `packages/player/src/player/live-follow-controller.ts:430`. Proposed direction: enforce cumulative limits after every decoded segment and discard irrelevant tabs immediately.

6. **Malformed incremental replay events can stall an authorized viewer.** Several rrweb variants only validate the source number, leaving malformed interaction types and movement positions to throw or create invalid timer work inside rrweb. Evidence: `packages/player/src/worker-core.ts:344` and `packages/player/src/decode-worker-source.ts:327`. Proposed direction: use a strict discriminated schema for every supported incremental source in both worker implementations and contain rrweb playback failures.

7. **Dynamic stylesheet changes inside blocked subtrees can leak private data.** Stylesheet observers emit raw rules and property values without checking `rr-block` or `data-orange-block` ancestors. Evidence: `packages/rrweb-fork/src/vendor/rrweb/record/observer.ts:598`. Proposed direction: carry the block selector into all stylesheet observers, check the owner/host at emission time, and test dynamic rules, replacements, and property changes.

## Environment-dependent security risk

1. **Private analytics reports follow an existing destination symlink.** `writeFile()` runs before `chmod()`, so an attacker who can write in the report directory could redirect or briefly expose the report. Evidence: `scripts/analytics/acceptance-lib.mjs:162`. Exploitability depends on deployment and CI workspace permissions. Proposed direction: verify those permissions, then use the existing no-follow private-file path with ancestor checks and atomic replacement.

## Scanner false positive

DeepSec reported that a failed `setAlarm()` could leave the earlier batch and state writes committed, allowing a duplicate retry to acknowledge a session with no lifecycle alarm. In this SQLite-backed Durable Object, the batch insert, state write, and alarm write have no intervening `await`; [Cloudflare documents that such writes are coalesced into one atomic implicit transaction](https://developers.cloudflare.com/durable-objects/best-practices/rules-of-durable-objects/). A defensive duplicate-path alarm check may still be useful, but the reported partial-commit sequence is not valid for this runtime.

## Remediation and revalidation — 2026-07-17

All eight security-relevant items above are fixed in the current working tree:

1. Public-page writes now require the revision captured when the dialog opened. The Worker uses an atomic compare-and-swap plus a per-request mutation token, returns `409 public_page_settings_changed` for stale writes, and never retries a stale draft automatically.
2. A live-ticket nonce is consumed before any snapshot work begins.
3. Finalization reads metadata in keyset pages and applies fixed segment, checkpoint, batch, byte, and sidecar limits. Oversized but valid recordings still finalize without full analytics instead of exhausting memory.
4. Migration `0022_accepted_usage_reservations.sql` adds a monotonic accepted-byte ledger. When enabled, accepted storage is reserved before acknowledgement and is moved into finalized quota accounting atomically.
5. Live-history discovery and playback enforce event, byte, and batch limits while decoding, and discard unrelated tabs before retaining their data.
6. Recorded and live replay paths share one strict incremental-event validator. It checks finite values, movement structure and ordering, and timestamps against trusted segment bounds; invalid data cannot enter rrweb playback.
7. Stylesheet mutation capture checks blocked owners, hosts, and adopters at emission time, including constructed sheets in documents, shadow roots, and iframes.
8. Private reports now validate the complete ancestor chain, require safe ownership and permissions, use no-follow exclusive file descriptors, publish once with a hard link, sync the file and directory, and bind success or failure receipts to the prepared report hash.
