## 1. PresenceRegistry DO (new)

New DO class `PresenceRegistry` in `apps/worker/src/do/` (follow the session-recorder/session-logic split pattern for testability). One instance per project via `idFromName(projectId)`. SQLite storage:

- `presence(session_id TEXT PRIMARY KEY, started_at INTEGER NOT NULL, last_seen INTEGER NOT NULL, entry_url TEXT, country TEXT, city TEXT, browser TEXT, os TEXT, device TEXT)`
- `project_meta(k TEXT PRIMARY KEY, v TEXT)` — holds `first_event_at` (ms epoch as string) once set.

Internal endpoints (called by the session DO and API worker — keep the same internal-fetch style the codebase already uses for DO communication):

- **ping**: upsert a presence row (insert with meta on first ping, update `last_seen` after); sets `project_meta.first_event_at` if absent. Idempotent.
- **remove**: delete the session's row (finalize). Idempotent.
- **list**: return sessions with `last_seen >= now - PRESENCE_TTL_MS`; in the same request, DELETE rows older than the TTL (lazy eviction — this is the ONLY eviction mechanism; no alarms).
- **install-status**: return `{ firstEventAt: number | null }` from `project_meta`.

`PRESENCE_TTL_MS` default 60_000, overridable via the existing `TEST_TIMINGS` mechanism.

## 2. Session DO integration

- On session start (first accepted batch) send a ping carrying the session meta the DO already holds (entry url, cf-derived country/city/browser/os/device, started_at).
- Heartbeats: at most one ping per `PRESENCE_HEARTBEAT_MS` (default 20_000, TEST_TIMINGS-overridable), piggybacked on the appendBatch request path — NEVER from a timer. Track last-ping time in memory/DO state.
- On finalize send remove.
- Registry failures must NEVER fail or delay ingest: fire-and-forget via `ctx.waitUntil`, catch everything, record the failure as a field on the existing wide event (e.g. `presence_ping_error`).

## 3. API worker endpoints (all authz-checked like existing routes, wide event per request)

- `GET /api/v1/projects/:projectId/live` → `{ sessions: [{ session_id, started_at, last_seen, entry_url, country, city, browser, os, device, duration_ms }] }` from the registry's list. `duration_ms = now - started_at`.
- `GET /api/v1/projects/:projectId/config` → project config from its D1 control-plane table (source of truth, NOT the KV cache).
- `PUT /api/v1/projects/:projectId/config` → body validated with a zod schema added to `packages/shared/src/schemas.ts` (follow the existing `z.ZodType<T>`-annotated style). Config fields per ARCHITECTURE.md: origin allowlist, sampling rate, masking policy version + rules, capture toggles, quota state, jurisdiction. Writes: D1 first (prepared statement, version bump in the row), then the KV `CONFIG` entry the ingest hot path reads (same key format ingest uses — read the ingest auth code to match exactly). Response echoes the stored config + new version. Invalid body → 400 with error code, no partial writes.
- `GET /api/v1/projects/:projectId/install-status` → proxies the registry DO's install-status. This powers the dashboard's "waiting for first event" install verify.
- Resolve the existing TODO in the API live-WS path (`handler.ts` — "jurisdiction-pinned sessions need the project's jurisdiction from config"): fetch the project's jurisdiction from config (D1) and resolve the session DO id the same way ingest does, so jurisdiction-pinned sessions connect to the correct DO.

## 5. Tests (the project's established pattern: `unstable_dev` integration tests against guarded `/__test/*` routes with `DEV_TEST_ROUTES=1`; extend test routes as needed; timings via `TEST_TIMINGS`)

Cover at minimum:

- ping → live list shows the session with meta; heartbeat updates `last_seen`; remove → gone.
- TTL lazy eviction: stale entry disappears from list (and its row is deleted) without any alarm.
- Heartbeat throttling: rapid appendBatches produce at most one ping per window.
- Registry unavailable/error → ingest still succeeds.
- install-status: null before first event → timestamp after → unchanged by later sessions.
- config PUT: validation failure 400; success bumps version, D1 row and KV entry both updated and consistent; GET returns D1 truth.
- live/config/install-status endpoints reject a missing or invalid Better Auth session (401) and reject users outside the project membership (403).
- Idempotency: duplicate pings/removes are safe.
