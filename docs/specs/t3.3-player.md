# T3.3 — `packages/player`: replay engine

Read `ARCHITECTURE.md` (§2 wire protocol, §3 segment format ORS1 + zero-server-decompression, §5 Playback) and `PLAN.md` (ground rules, testing strategy) first. Scope: create `packages/player/**` ONLY, plus its workspace registration if the root config needs it. Do NOT touch apps/, other packages, or the design mocks. Do NOT run git.

The player is a **headless, embeddable engine** consumed by the dashboard (T3.5 builds the visible controls per the design language). It exposes an imperative API + event emitter; the only DOM it owns is the replay viewport it is given and canvas/DOM overlays inside it.

## Package shape

- `packages/player` mirrors the other packages: tsdown build, `sideEffects: false`, subpath-clean exports, `vp check`/`vp test` wired, happy-dom tests.
- **Dependency**: upstream `rrweb` (npm, latest 2.x) for its `Replayer` ONLY — justified by PLAN.md risk register ("replay side uses upstream npm in the player; the vendored fork is capture-side only"). No other new runtime deps. `@orange-replay/shared` for wire/types.

## Data plane

1. **Manifest loader**: `loadSession(api, { projectId, sessionId })` fetches the manifest (shape: `SessionManifest` in `packages/shared/src/types.ts` — `segments: SegmentRef[]`, `timeline: IndexEvent[]`, `counts`, `startedAt/endedAt/durationMs`). The `timeline` array alone must be enough to render an **instant timeline** (activity ticks, error/click/rage markers, duration) before any segment bytes arrive.
2. **Segment fetch + ORS1 slicing**: segments come as ORS1 binaries from the API segment route. Parse with `parseSegment`/`segmentBatch` from `@orange-replay/shared/wire` — never re-implement the codec. Each contained batch is a gzip (or FLAG_UNCOMPRESSED) rrweb-event JSON payload.
3. **Decode Web Worker**: all gunzip (`DecompressionStream`) + JSON.parse happens in a dedicated Web Worker (follow the SDK's worker bundling pattern in `packages/sdk` — inline-able so CSP-degraded hosts still work with a synchronous fallback). Main thread receives parsed rrweb event arrays via transferable/structured clone. The main thread NEVER decompresses.
4. **Prefetch strategy**: fetch segment n, start playback as soon as its events land, prefetch n+1 while playing; seeking to a time maps to its segment (seek-by-segment) and re-anchors prefetch.

## Playback engine

- Wrap upstream `rrweb.Replayer` (virtual-dom iframe mode) behind `OrangePlayer`:
  - `new OrangePlayer(container, { api, projectId, sessionId })`; private dashboard calls use the same-origin Better Auth session cookie, while `/demo` uses its anonymous read-only API context.
  - `play() / pause() / seek(ms) / setSpeed(x) / setSkipInactivity(bool) / follow() / destroy()`
  - events: `ready` (manifest loaded), `timeline` (instant timeline data), `progress` (current ms), `segment` (n loaded), `buffering`, `ended`, `live` (follow state), `error`.
- **Skip-inactivity**: gaps > 5s with no user events fast-forward at high speed (rrweb `skipInactive` where possible; ensure our timeline progress events stay monotonic).
- **Overlays** (drawn by us over the replay viewport, not injected into the replayed DOM):
  - **Cursor trail**: fading polyline following mouse positions (last ~1s of movement).
  - **Click ripples**: expanding ring at each click.
  - **Rage-click burst**: distinct multi-ring red burst when ≥3 clicks land within 600ms/24px (derive from event stream; constant with the sidecar's rage definition if one exists in shared constants).
  - Overlay colors/opacities as parameters with defaults matching the design tokens (amber #f5a623 playhead affinity, red #f4534e for rage) so the dashboard can pass exact values.
- **Live follow**: mint a short-lived live ticket through the normal REST route, then open the WebSocket at `/api/v1/projects/:p/sessions/:s/live?ticket=…`. Mint a fresh ticket for each reconnect. Incoming binary frames are wire ingest bodies: `decodeIngestBody` → payload → worker decode → append to replayer in follow mode (`play` at live edge; new events chase the edge). Reconnect with backoff; surface `live` state changes. Undici/browser differences: set `binaryType = "arraybuffer"`.

## Testing (happy-dom under `vp test`; no Playwright here — T3.7 covers browser e2e)

Pure-logic units, extracted into plain functions:

- timeline building from `IndexEvent[]` (ticks, markers, duration)
- time→segment mapping + prefetch window decisions
- skip-inactivity gap detection
- rage-click burst detection
- live-frame handling: decodeIngestBody → batch ordering by (tab, seq), duplicate-frame tolerance
- worker protocol messages (encode/decode round trip with a stubbed worker)

Mock the network (manifest/segments/WS) with fixtures built via `buildSegment`/`encodeIngestBody` from shared — wire-conformance by construction.
