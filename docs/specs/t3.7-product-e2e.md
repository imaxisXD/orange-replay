## 1. New spec file `tests/product.e2e.ts` — scenarios in order

1. **Record → list**: context A opens the demo site, performs a deterministic interaction script (clicks incl. the TypeError button, typing, scroll, page2 navigation), closes. 2. **Playback with effects**: click the row → detail. Assert: replayer iframe appears and its document contains known demo-site copy (e.g. "Warehouse stock timeline" — same assertion style as recording.e2e.ts uses on decoded payloads); press play (or programmatically via the page's exposed player if needed — prefer real UI clicks); assert the timeline shows ≥1 error marker element; assert the click-ripple/cursor overlay layer element exists in the player viewport during playback; assert the event sidebar lists the TypeError rows and clicking one moves the playhead (compare playhead style/position or current-time text before/after).
3. **Masking proof (dashboard-level)**: the replayer iframe document must NOT contain the password text typed into the masked field during recording (reuse the literal from the interaction script) — the list/detail surfaces must never leak it either.
4. **Live now**: context A opens the demo site and stays active (periodic small interactions to keep batches flowing); dashboard Live tab (context B) shows ≥1 live row with the demo entry page within ~15s; the row's elapsed time updates across a poll cycle.
5. **Two-context live watch**: with context A still recording, context B opens the session detail (from the Live row click) and the player enters follow mode: assert the LIVE indicator renders; drive a distinctive DOM change in context A (e.g. click "Add product row" adding a known text row) and assert that text appears inside the replayer iframe in context B within a reasonable window (≤20s). Then context A closes; after finalize the LIVE indicator clears or playback continues without error (assert no error alert).
