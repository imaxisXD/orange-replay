Read PLAN.md ground rule 7 and ARCHITECTURE.md §8. This is an audit-and-fix task over the WHOLE worker surface. Scope: `apps/worker/**`, `packages/shared/src/**` (schemas/constants only if a contract needs it), `packages/player/src/api.ts` + `apps/dashboard/src/lib/api.ts` (ticket client change in §2 only), plus tests. Do NOT run git. Keep every cost invariant (no timers in DOs, no hot-path writes, ingest never decompresses).

## 1. Audit checklist (fix anything failing; list PASS/FIXED per item in your report)

- **Authz on every route**: enumerate every route in `apps/worker/src/index.ts` + `api/handler.ts` + `ingest/handler.ts`; assert each is either (a) bearer-authenticated, (b) write-key authenticated (ingest), (c) explicitly public (`/api/v1/health` only), or (d) DEV_TEST_ROUTES-gated. Any other route = finding.
- **R2 key validation**: every R2 read/write derives keys via the validated helpers (`sessionPrefix`/`manifestKey`/segment regex); no user-controlled string reaches R2 without id validation. Audit the test routes too (gated, but still validate).
- **Input caps**: ingest body cap enforced pre-read (readBodyCapped); index JSON size cap; batch count caps; config PUT body size cap (add one if missing — e.g. 64KB via content-length check before json()); presence ping body caps (registry validates types — add length caps for strings if missing).
- **Quota-flip path**: verify quota_state=exceeded → ingest drops with 202 `dropped` outcome and no DO write; add/verify an integration test.
- **Rate limiting**: verify what exists; if nothing beyond size caps, add a MINIMAL in-DO guard on the session recorder: reject appends when a session exceeds a sane batch rate (e.g. >30 appends per 10s window, in-memory counter, no storage writes, 429 with a `rate_limited` wide-event outcome) — protects against a runaway/hostile SDK without new infrastructure. Constants in shared; TEST_TIMINGS-overridable for tests.
- **Security headers**: all API JSON responses get `x-content-type-options: nosniff` and `referrer-policy: no-referrer`; segment/manifest responses additionally keep their cache headers; CORS on ingest remains exact-origin per config (verify no `*` leak when allowedOrigins is specific).
- **Error hygiene**: error responses never echo internal exception messages to clients on 5xx (generic body; details stay in wide events). Audit and fix any leaky ones.

## 2. Live WS tickets (closes the pinned P3 security note)

Replace the `?token=<master token>` live-WS transport with short-lived signed tickets:

- `POST /api/v1/projects/:id/sessions/:sid/live-ticket` (bearer auth like siblings) → `{ ticket, expiresAt }`. Ticket = HMAC-SHA256 over `projectId:sessionId:exp` keyed by the API token (WebCrypto, timing-safe verify), encoded compactly (base64url of `exp.sig`). TTL 60s, single purpose.
- The live WS route accepts `?ticket=` (verify signature + expiry + ids) and REJECTS `?token=` (remove the old fallback; update the code comment).
- Player: `follow()` fetches a ticket via the authenticated REST client just before connecting (and on each reconnect — reconnects must mint a fresh ticket since TTL is 60s). Dashboard api client passes through unchanged (player owns minting via its existing options.api + token).
- Tests: ticket mint requires auth; expired/garbage/cross-session tickets rejected (401); happy path connects; reconnect after expiry works (mint again); `?token=` no longer accepted.
- Wide events: live connect event gains `auth: "ticket"`; no ticket material logged.
